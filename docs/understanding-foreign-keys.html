01What is a Database Table?

Think of it as a spreadsheet with strict rules. Each row is one "thing" and each column describes a property of that thing.

📋

The Spreadsheet Metaphor

Imagine an Excel sheet where every column has a fixed name and data type. You can't put a person's name in the "age" column. A database table is exactly this — a structured grid of data with rules.

Here's a simple table that stores employees at a company:

👤 employees
idnameroledepartment
1AliceEngineerBackend
2BobDesignerFrontend
3CharlieManagerBackend

Each row is one employee. Each column (id, name, role, department) is a property. Simple!

02What is a Primary Key?

Every row needs a unique ID badge. No two rows can share the same one. This is the Primary Key.

📛

The ID Badge Metaphor

At a conference, everyone gets a badge with a unique number. Even if two people are named "Alice", their badge numbers are different. That badge number is the Primary Key — it uniquely identifies each person.

👤 employees — PK = id
id 🔑namerole
1 ← unique!AliceEngineer
2 ← unique!BobDesigner
3 ← unique!CharlieManager
🔑 Key Rule

A Primary Key must be unique and never empty. The database enforces this — if you try to insert a row with id=1 when id=1 already exists, the database rejects it.

03What is a Foreign Key?

A Foreign Key is a column that points to a Primary Key in another table. It's a reference — a link between two tables.

👉

The Pointer Metaphor

Imagine each employee has a card that says "My manager is badge #3." That card is a Foreign Key — it doesn't contain the manager's info, it just points to the manager's row using their ID badge number.

Let's add a manager_id column. It points to another employee's id:

👤 employees — PK id    FK manager_id → employees.id
id 🔑namerolemanager_id 👉
1AliceEngineer3 → Charlie
2BobDesigner3 → Charlie
3CharlieManagerNULL (no manager)
🛡️ The FK Rule

A Foreign Key value must exist in the referenced table. If you try to set manager_id = 99 but no employee has id = 99, the database says "No! That person doesn't exist!" and rejects the operation.

04The Self-Referential FK

When a table's Foreign Key points back to itself. The employees table above is exactly this — manager_id points to another row in the same table.

🌳

The Family Tree Metaphor

In a family tree, each person has a "parent" field. But the parent is also a person in the same list! The list points to itself. That's a self-referential Foreign Key.

This creates a tree structure — like an org chart:

id=3, manager_id=NULL
Charlie (Manager)
id=1, manager_id=3
Alice (Engineer)
id=2, manager_id=3
Bob (Designer)

This is exactly how Code Palace stores code symbols. A class contains methods. The method's parent_id points to the class — both live in the same symbols table.

05The DuckDB Snapshot Problem

Here's where things get tricky. DuckDB has a quirk: when checking Foreign Keys, it looks at a frozen snapshot of the table — not the live, current data.

📸

The Photograph Metaphor

Imagine a security guard who checks IDs. But instead of looking at the current list of employees, they use a photograph of the list taken 5 minutes ago. If someone was just hired, they're on the real list but NOT in the photo. The guard says "I don't see you in my photo — access denied!"

06The Fix: Insert in Order

The solution is simple: insert the parent first, then insert the child with its parent_id already set. Never update parent_id after the fact.

✅

The Correct Approach

Instead of hiring everyone and then assigning managers later, hire the manager first, get their badge number, then hire employees with "my manager is badge #X" already filled in on their form.

✅ Interactive: The Correct Way

This time, we insert the parent class FIRST, get its ID, then insert the method WITH parent_id already set.

📦 symbols
symbol_id 🔑namekindparent_id 👉
💡 Why This Works

When the method is inserted with parent_id=1, DuckDB takes a snapshot and checks: "Does symbol_id=1 exist?" YES — we already inserted it in step 1! The FK check passes because the parent was there before we referenced it.

07How This Applies to Code Palace

Code Palace indexes your codebase into a database. Every function, class, and method becomes a row in the symbols table — with parent_id pointing to its containing symbol.

Consider this Python code:

class UserService:
    """Handles user operations."""

    def create_user(self, name):
        ...

    def delete_user(self, user_id):
        ...

Code Palace stores this as:

📦 symbols — PK symbol_id    FK parent_id → symbols.symbol_id
symbol_id 🔑namekindparent_id 👉file_id
1 UserService class NULL 42
2 create_user method 1 → UserService 42
3 delete_user method 1 → UserService 42

The tree structure:

symbol_id=1, parent_id=NULL
class UserService
symbol_id=2, parent_id=1
method create_user
symbol_id=3, parent_id=1
method delete_user
🚨 What We Tried (and Failed)

We tried to speed things up by inserting ALL symbols at once with parent_id=NULL, then updating parent_id afterward. DuckDB's snapshot-based FK checker rejected the UPDATE — it couldn't "see" the parent rows it needed to reference.

✅ What We Do Instead

Code Palace inserts symbols in tree-sitter order — parent classes are parsed before their methods. So by the time a method is inserted with parent_id=1, the class with symbol_id=1 already exists. The FK check passes every time.

Recap: The Three Key Lessons

🔑

Primary Key

A unique ID badge for every row. No duplicates allowed.

👉

Foreign Key

A pointer to another row's ID. The pointed-to row MUST exist.

📸

DuckDB Snapshot

DuckDB checks FKs against a photo of the past, not the present. Insert parents before children!
